## Build & Flash

```bash
# Full workflow (compile + upload + monitor)
./build_and_monitor.sh

# Compile only — no upload
./build_and_monitor.sh --skip-upload

# Upload only — skip recompile (use cached build)
./build_and_monitor.sh --monitor-only   # skips both, just monitors

# Explicit port
./build_and_monitor.sh -p COM5
```

Raw arduino-cli equivalents (if you want to run steps manually):
```bash
CLI="/c/Program Files/Arduino CLI/arduino-cli.exe"

# Compile
"$CLI" compile --fqbn UNIHIKER:esp32:k10 \
  --build-property "build.cdc_on_boot=1" \
  --libraries ./lib .

# Upload
"$CLI" upload -p COM9 --fqbn UNIHIKER:esp32:k10 .

# Monitor (requires interactive terminal — won't work headless)
"$CLI" monitor -p COM9 --config baudrate=115200
```

## Upload Web Files to SD Card

The firmware exposes a `/upload` endpoint. Use this to push the `data/` directory
to the ESP32's SD card without physically removing it:

```bash
# Upload all web assets (JS, HTML, CSS)
for f in data/*.js data/*.html data/*.css; do
  curl --limit-rate 50K -F "file=@$f" "http://192.168.4.72/upload?dest=/data/$(basename $f)"
  echo
  sleep 0.5   # let ESP32 WiFi RX buffer drain between files
done

# Upload survivor portraits (only needed when images change)
for f in data/img/survivors/*.jpg; do
  curl --limit-rate 50K -F "file=@$f" "http://192.168.4.72/upload?dest=/data/img/survivors/$(basename $f)"
  echo
  sleep 0.5
done
```

- IP is `192.168.4.72` (confirm with `arduino-cli board list` or check your router)
- Server responds `OK` per file on success
- `--limit-rate 50K` prevents WiFi RX buffer corruption on large files (>12KB)
- **Always include `*.css` in the upload** — a stale CSS file causes overlays to lose `position:fixed` and cards to lose all styling, even if JS files are fine
- Reboot the ESP32 after uploading so it reloads files from SD into PSRAM
- New JS files also need entries in the `WEB_FILES[]` array in `Esp32HexMapCrawl.ino`

## Serial Monitoring — Current Status

The `arduino-cli monitor` command **requires an interactive TTY** — it exits immediately when run from a script or subprocess. This is a known limitation.

Workarounds we've tried so far and their status:

| Method | Result |
|---|---|
| `arduino-cli monitor` in script | Exits immediately (no TTY) |
| PowerShell `SerialPort` | 0 bytes — board silent in boot screen |
| Python pyserial | 0 bytes — same reason |
| DTR toggle reset | Put board into download mode instead of run |
| `cdc_on_boot=1` compile flag | Enabled, re-flashed ✓ — but still 0 bytes |

The remaining issue is **timing**: setup() serial output fires in the ~2s window right after flash before the monitor connects, and loop() never runs because the **boot screen is blocking waiting for a button press on the device**.
